# 完整 AI 代理移植计划：slab.rs ← codex-rs（2026-05-20）

本文是对当时移植工作的结构化记录，保留目标、分步范围和约束，便于后续审计与对照。

## 现状（当时）

- `slab-agent` 已具备编排内核能力（控制、线程、转向、钩子、工具特性）。
- `slab-agent-tools` 已提供基础工具：`shell`、`read_file`、`write_file`、`list_dir`、`grep`、`fs_watch`。
- 缺口：沙盒、MCP、Git 工具、`apply_patch`、审批/执行策略尚未完整落地。

## 步骤 1：沙盒层

参考：`codex-rs/sandboxing`、`codex-rs/linux-sandbox`、`codex-rs/process-hardening`

- `crates/slab-sandboxing`：平台无关核心能力。
- `crates/slab-linux-sandbox`：仅 Linux（`#[cfg(target_os = "linux")]`）。
- `crates/slab-windows-sandbox`：Windows 对应实现。

关键约束：

- 不引入遥测副作用。
- 可观测性仅通过 `tracing` 事件与跨度表达。

## 步骤 2：MCP 协议引擎

参考：`codex-rs/codex-mcp`、`codex-rs/mcp-server`、`codex-rs/rmcp-client`

目标模块：`crates/slab-mcp`（客户端 + 服务端）

客户端能力（`slab_mcp::client`）：

- `McpClient`：连接 MCP 服务器（stdio 子进程或 SSE/HTTP 端点）。
- 传输层：`StdioTransport`、`SseTransport`。
- JSON-RPC 2.0 消息：`McpRequest`、`McpResponse`、`McpNotification`。
- 发现：`list_tools() -> Vec<McpToolSpec>`。
- 调用：`call_tool(name, args) -> McpToolResult`。
- 生命周期：`initialize()`、`ping()`、优雅关闭。
- 可将远程工具注册到 `ToolRouter`（`McpProxyTool` 适配）。

服务端能力（`slab_mcp::server`）：

- 暴露已注册工具给 MCP 客户端。
- stdio 传输，兼容 `npx @modelcontextprotocol/inspector`。
- 处理 `initialize`、`tools/list`、`tools/call`、`ping`。
- 返回 MCP 规范的内容结构。

配置模型（`slab_mcp::config`）：

- `McpServerLauncher { name, command, args, env }`
- `McpClientConfig`（启动时连接的服务器列表）

依赖约束：

- 不引入 `codex-otel`、`codex-analytics`。
- 使用 `tokio`、`serde`、`serde_json`、`tracing`、`thiserror`、`async-trait` 等通用组件。

## 步骤 3：文件与 Git 能力

- `crates/slab-file-watcher`：替换旧的 Git 源依赖方式，支持离线开发。
- `crates/slab-file-system`：承接文件工具能力。
- `crates/slab-git`：提供 Git 状态、差异、提交等能力。

## 步骤 4：独立 Shell 执行器

参考：`codex-rs/shell-command`

目标模块：`crates/slab-shell-command`

核心结构：

- `ShellExecutor`：执行策略、工作区根、可选 `SandboxDriver`。
- `ShellCommand`：`{ command, timeout_secs, env }`。
- `ShellOutput`：`{ stdout, stderr, exit_code, timed_out }`。

能力要求：

- 危险命令检查（含 `rm/dd`、`mkfs.*`、权限泛化等模式）。
- 执行策略检查（允许/需审批/拒绝）。
- 可路由到沙盒驱动，而非直连 `tokio::process::Command`。
- PTY 支持、信号转发、输出截断。
- `slab-agent-tools/src/shell.rs` 委托给 `ShellExecutor`。

## 步骤 5：扩展 slab-agent-tools

新增工具目标：

- `apply_patch`（基于 `slab-file-system`）
- `git_status`、`git_diff`、`git_commit`（基于 `slab-git`）
- `mcp_call`（基于 `slab-mcp`）

`register_all_tools` 目标签名：

```rust
register_all_tools(
    router,
    shell_policy: ShellPolicy,
    sandbox_driver: Option<Arc<dyn SandboxDriver>>,
    workspace_root: Option<PathBuf>,
    mcp_client: Option<Arc<McpClient>>,
    git_tools: bool,
)
```

注册行为：

- 基础工具：`echo`、`shell`、`read_file`、`write_file`、`list_dir`、`grep`、`fs_watch`、`apply_patch`
- `git_tools=true` 时注册 `git_status`、`git_diff`、`git_commit`
- 提供 `mcp_client` 时自动发现并代理远程 MCP 工具

## 依赖关系图（目标态）

```text
slab-app-core
  └── slab-agent
        └── slab-agent-tools
              ├── slab-shell-command
              │     └── slab-sandboxing
              │           ├── slab-linux-sandbox   (cfg linux)
              │           └── slab-windows-sandbox (cfg windows)
              ├── slab-file-watcher
              ├── slab-file-system
              ├── slab-git
              └── slab-mcp
```

## 全局约束

- 结构化可观测性走 `tracing`，由宿主注入 `tracing_subscriber`。
- 新 crate 不直接向分析端点发 HTTP。
- 新 crate 不做文件型部署日志。
- 不引入 `codex-rollout`、`codex-analytics`、`codex-otel`。

## 实施顺序建议（当时）

1. `slab-file-watcher`
2. `slab-sandboxing` + `slab-linux-sandbox` + `slab-windows-sandbox`
3. `slab-shell-command`
4. `slab-file-system`
5. `slab-git`
6. `slab-mcp`
7. `slab-agent-tools` 整体接线

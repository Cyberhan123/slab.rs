
# slab.rs Bazel 构建升级方案 

## Summary

将现有 Bazel-first 但仍依赖 Cargo 包装目标的构建体系，升级为 Bazel-native CI/CD 与发布体系：Cargo 继续作为 Rust 依赖、版本、features 的单一事实来源；Bazel 负责 hermetic toolchains、依赖解析、构建、测试、clippy、跨平台发布、产物 staging 和缓存。

当前仓库已有 `MODULE.bazel`、`.bazelrc`、多数 crate 的 `BUILD.bazel`、Bazel wrapper scripts 和 CI 基础；本方案按允许破坏性变更处理，替换掉 Cargo wrapper、嵌套 Bazel 调用、cargo-make release 残留，并把 Rust/LLVM/platform/toolchain/patch/staging 收敛到 Bazel。

## Key Changes

- `MODULE.bazel` 改为 `rules_rs 0.0.58` + `rules_rust` Rust toolchain 承载模式，保留 Cargo 输入为根 `Cargo.toml` / `Cargo.lock`，统一生成 `@crates`。
- 加入 `llvm 0.7.1`、`rules_cc 0.2.16`、`platforms 1.0.0`，建立 hermetic LLVM/Clang 配置；平台目标覆盖 Linux/macOS/Windows 的 `x86_64` 与 `aarch64`。
- 将 `supported_platform_triples` 扩展为用户指定的 6 个 triples，并补齐平台约束与 release config：`linux-x86_64`、`linux-aarch64`、`macos-x86_64`、`macos-aarch64`、`windows-x86_64-gnullvm`、`windows-aarch64-gnullvm`。
- `.bazelrc` 增加 `--enable_runfiles`、remote cache 预留 config、release/cross config、平台 config 和 CI config；保留 Windows 短 output base、symlink 和 strict action env。
- 新增 `justfile` 只作为薄入口：`just bazel-lock-check` 执行 `bazelisk mod deps --lockfile_mode=error`；不引入新的并行编排层。

## Implementation Changes

- 重写 `tools/bazel/rust.bzl`，新增唯一宏：
  ```starlark
  def slab_rust_crate(
          name,
          crate_name,
          crate_features = [],
          build_script_enabled = True,
          compile_data = [],
          test_data_extra = [],
          test_tags = [],
          deps_extra = []):
      """统一的 Rust crate 封装，自动生成 library + binary + tests"""
  ```
- `slab_rust_crate` 自动根据本包源码生成 `:<name>_lib`、主 binary、`rust_test` targets；`deps_extra` 只用于内部 crate 或特殊 data/link deps，第三方依赖来自 `@crates`。
- 将现有 `slab_rust_library` / `slab_rust_binary` BUILD 文件迁移到 `slab_rust_crate`，优先覆盖核心路径：`slab-server`、`slab-runtime`、`slab-js-runtime`、`slab-python-runtime`、`slab-app-core`、`slab-proto`、`slab-agent*`、`slab-types`、`slab-config`、`slab-utils`、ML sys crates、installer。
- `crate.from_cargo` 负责三方 crate 自动目标；`crate.annotation` 只保留必要补丁：vendored Rust patch crates、`ffmpeg-sys-next`、`pyo3-build-config`、`v8`、`wasmtime-internal-versioned-export-macros`、含 build script/FFI 数据的 crates。
- 将 `vendor/slab-artifacts.toml` 纳入 Bazel artifact repository/staging 规则输入，统一管理 `ggml`、`llama`、`whisper`、`diffusion`、`ffmpeg` 的平台资产。
- 将 `crates/slab-proto/build.rs` 的 proto 生成迁移到 Bazel proto/Rust proto target；`build.rs` 只保留 Cargo 开发路径需要的兼容 glue。
- 将 `*-sys` bindgen/FFI 路径改为 Bazel toolchain + data 输入，移除对宿主环境 `PKG_CONFIG_PATH` / 手工下载目录的依赖。
- 替换 `tools/cargo/*` 的 workspace check/test/clippy：CI 中 `check:rust`、`test:rust`、`lint:rust` 全部使用 Bazel-native Rust targets，不再执行 `cargo check/test/clippy`。
- 替换 `bin/slab-app:dev`、`bundle_debug`、`bundle_release_windows` 中的嵌套 `bazelisk run //tools/cargo:build_sidecars`，改为直接依赖 Bazel-built sidecar targets 并通过 runfiles/staging actions 交给 Tauri。
- Release workflow 删除 `cargo make release`、Rust toolchain install、手工 FFmpeg bootstrap；改为 Bazel matrix 发布目标，并上传 Bazel staging 输出。

## CI/CD And Release

- CI matrix 扩展到 Ubuntu、macOS、Windows；每个平台跑 `bazelisk mod deps --lockfile_mode=error`、`bazelisk query //...`、`bazelisk build //...`、`bazelisk test //...`、Bazel clippy target。
- Release workflow 增加 6 个平台发布 target；macOS SDK 和 Windows MinGW/gnullvm 工具链通过 Bazel toolchain/config 管理。
- 多平台二进制发布产物由 Bazel target 命名并输出到统一 staging layout，例如 `bazel-bin/dist/<platform>/...`。
- `package.json` 保持薄 wrapper，日常开发可保留 Cargo 命令但不进入 CI/CD；所有稳定检查和发布入口走 Bazel。

## Test Plan

- 依赖/锁文件：`just bazel-lock-check`，`bazelisk mod deps --lockfile_mode=error`。
- 图解析：`bazelisk query //...` 必须稳定完成；当前 60 秒超时需作为第一阶段问题修复。
- 构建一致性：对主要二进制执行 `bazelisk build` 与 `cargo build --workspace` 对照，确认 binary 名称、features、runtime assets 和 sidecar staging 一致。
- 全量验证：`bazelisk build //...`、`bazelisk test //...`、Bazel clippy target、frontend Bazel build target。
- 发布验证：每个平台 release target 输出可上传资产；Windows 验证 Tauri externalBin、runtime libs、FFmpeg DLL、offline installer CAB；macOS 验证 `.app`/`.dmg`；Linux 验证可执行文件和运行时资源目录。
- Contract drift：后端 API 变更仍用 Bazel-backed `bun run gen:api`，并检查 `packages/api/src/v1.d.ts` 无未提交漂移。

## Assumptions

- 采用用户指定版本：`rules_rs 0.0.58`、Rust `1.95.0`、edition `2024`、LLVM `0.7.1`、`rules_cc 0.2.16`。
- Cargo 仍是 Rust 依赖真源；Bazel 不手写三方 Rust 依赖列表。
- 允许破坏性替换现有 Bazel/Cargo wrapper、CI release workflow 和 BUILD 宏签名。
- 可选保留 Cargo 日常开发路径，但 Cargo 不再作为 CI/CD 或 release 的执行路径。
- Tauri CLI、Bun/Vite 可以继续作为底层打包/前端构建工具，但必须由 Bazel target 调度和 staging。
